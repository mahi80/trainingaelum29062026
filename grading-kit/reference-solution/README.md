## Minimal approach (per lane)

The reference favours **transparent heuristics over big models** wherever that
still clears Target. The intent is that a candidate who does the obvious correct
thing lands at or just above these numbers.

### 1. `router_planner` â€” keyword/heuristic routing
Weighted keyword-cue scoring over `{sql, sparql, vector, underwrite}`, with a
small priority weighting so specialised lanes win ties against the generic SQL
lane. Two co-active lanes trigger a `hybrid` plan. Returns a valid `route`, an
ordered node `plan`, and a routing `confidence` (logistic of the winner/runner-up
gap). **No LLM required** â€” keeps the harness deterministic. A strong candidate
swaps in an LLM JSON classifier; the heuristic is the calibration floor.

### 2. `schema_linker` â€” trigram/keyword over a schema catalog
Blends questionâ†”card keyword overlap with character-trigram (pg_trgm-style)
similarity against a trimmed offline schema catalog, keeps the top-5 tables, and
emits `linked_columns` + naive `join_hints`. Works with **no DB and no
embeddings**. The full candidate node does pgvector kNN over
`app.schema_embedding` + OpenMetadata FK expansion â€” that is the Stretch path.

### 3. `nl2sql_generator` â€” prompt + `get_llm().complete(as_json=True)`
Builds a constrained prompt from the linked sub-schema (SELECT-only, linked
tables/columns only, JSON output) and calls the project LLM at `temperature=0`.
On any LLM failure it falls back to a safe templated `SELECT count(*)` over the
primary linked table and sets `degraded=True`. Validation/repair + execution
stay in the provided `sql_validator_repair` / `sql_executor` nodes.

### 4. `vector_retriever` â€” pgvector `<=>` over `doc.document_chunk`
Embeds the question with the project LLM, then runs a cosine-distance
`ORDER BY embedding <=> %s::vector LIMIT 40` over the HNSW index, returning
`candidates`. This is the **dense half** of the intended hybrid retriever; it is
enough to anchor recall. The Target/Stretch candidate adds Postgres FTS + RRF
fusion. Degrades to an empty candidate list when `DATABASE_URL` is unset.

### 5. `reranker` â€” cross-encoder if available, else identity
Uses a `sentence-transformers` `CrossEncoder` (`bge-reranker-v2-m3`) when the
library/model are present, otherwise preserves the retriever's cosine order
(identity). Emits top-6 `contexts` with `rerank_score` and a `retrieval_scores`
summary so the **nDCG lift over no-rerank** is measurable either way.

### 6. `underwriting_scorer` â€” LightGBM if present, else logistic
Loads `artifacts/pd_lgbm.txt` (env `MODEL_PATH`) when LightGBM is installed and
the artifact exists; otherwise uses a transparent, monotonic **logistic
fallback** with hand-tuned coefficients. Buckets PD into
`approve (â‰¤0.15) / refer / deny (â‰Ą0.45)`, attaches `pd_confidence` (distance to
the nearest boundary), and ranks **adverse-action `reason_codes`** â€” SHAP-derived
with a model, contribution-derived for the logistic path. Reasons use Reg B /
ECOA-style language. The serve-time `_featurize` is shared with the trainer to
guarantee **no train/serve skew**.

### Nodes left to the provided stubs / not in this reference
`memory`, `sql_executor`, `explainer_citation` ship working in `src.nodes`.
`aggregator_critic`, `sql_validator_repair`, `hitl_gate`,
`sparql_ontology_agent`, `policy_compliance_checker` are **out of scope for this
light reference** â€” `sparql_ontology_agent` and `policy_compliance_checker` are
Stretch, and the rest are scored from the candidate's own implementation.

## Regenerating the trained PD artifact

```bash
pip install -r challenge-pack/requirements.txt
pip install -r grading-kit/reference-solution/requirements.txt   # optional extras
export DATABASE_URL=postgresql://app:app@localhost:5432/autoloan
python grading-kit/reference-solution/train_underwriting.py \
    --dsn "$DATABASE_URL" \
    --out grading-kit/reference-solution/artifacts
# -> artifacts/pd_lgbm.txt  (or pd_logistic.json on sklearn fallback)
# -> artifacts/adverse_action_map.json
```

The label is `defaulted` (90+ DPD or charged-off); when the seed lacks default
history it weak-labels from `underwriting_decision.pd_score`. `_featurize` is
imported from `nodes_ref` so the trained model and the serving node share one
feature space.
